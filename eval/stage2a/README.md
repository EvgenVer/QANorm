# Stage 2A Eval Set

Этот каталог содержит локальный eval-набор для приемки `Stage 2A MVP`.

## Цели

- проверять качество `document discovery` для инженерных вопросов без явной нормы;
- проверять retrieval по вопросам с упоминанием документа, но без точного locator;
- проверять поддержку сокращений и "грязного" пользовательского ввода;
- отдельно держать небольшой hidden diagnostic slice для технической проверки locator retrieval.

## Сценарии

- `no_explicit_norm_engineering`
  Основной пользовательский сценарий. Вопрос инженерный, но документ не назван.
- `explicit_document_without_locator`
  Пользователь упоминает документ, но не называет точный пункт.
- `compact_alias_dirty_input`
  Вопрос содержит сокращения, слитные коды и разговорный ввод.
- `ambiguous_scenario`
  Вопрос слишком общий; хороший ответ может быть `clarify` или аккуратный `partial`.
- `diagnostic_locator_hidden`
  Небольшой технический срез для проверки locator-aware retrieval. Это не основной acceptance signal.

## Формат `questions.jsonl`

Каждая строка — отдельный JSON-объект:

- `id`
  Стабильный идентификатор вопроса.
- `query`
  Текст пользовательского запроса.
- `scenario`
  Один из сценариев выше.
- `expected_mode`
  Ожидаемый режим ответа: `direct`, `partial` или `clarify`.
- `expected_documents`
  Список документов, которые считаются релевантными для ответа.
- `expected_locators`
  Ожидаемые локаторы. Для основного пользовательского набора чаще пусто.
- `must_include_terms`
  Нормализованные смысловые термины, которые должны появиться в ответе или evidence.
- `must_not_use_documents`
  Документы, использование которых считается ошибкой.
- `notes`
  Короткое пояснение для ручной проверки.

## Правила набора

- основной набор не должен состоять из вопросов с точными пунктами, таблицами и формулами;
- точные locator-кейсы держатся только в `diagnostic_locator_hidden`;
- набор должен быть смещен в сторону реальных инженерных вопросов, где система сама определяет, что искать;
- приоритет — проверять актуальные нормы, а не способность повторять старые документы и пособия.

## Target Thresholds

Для `Stage 2A MVP` фиксируются следующие целевые метрики полного прогона:

- `document hit@3 >= 0.85`
- `locator hit@5 >= 0.70`
- `grounded answer rate >= 0.95`
- `unsupported claim rate <= 0.05`
- `partial answer rate <= 0.25`
- `expected mode match rate >= 0.75`
- `wrong document rate <= 0.10`

Эти пороги используются как рабочие acceptance thresholds для `Block H` и для следующей итерации улучшений retrieval/agent runtime.
