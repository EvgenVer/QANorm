# Stage 2A Streamlit Manual Acceptance Checklist

## 1. Подготовка

1. Запустить UI:
   `streamlit run src/qanorm/stage2a/ui/app.py`
2. Убедиться, что:
   - приложение стартует без traceback;
   - открывается локальный URL Streamlit;
   - на странице видны заголовок `QANorm Stage 2A` и поле ввода вопроса.

Ожидаемый результат:
- UI доступен;
- пустой экран чата отображается корректно;
- ошибок подключения к БД или Gemini нет.

## 2. Базовый happy path с явной нормой

Шаги:
1. Ввести вопрос:
   `Что сказано в СП 63.13330.2018 пункт 5.1?`
2. Дождаться завершения ответа.

Проверить:
- показан непустой ответ;
- под ответом отображается режим ответа;
- открыт блок `Evidence`;
- в `Evidence` есть хотя бы один `evidence_id`;
- у evidence показаны локатор, заголовок и текст фрагмента;
- если включен debug, есть блок `Debug Trace`.

Ожидаемый результат:
- режим ответа обычно `direct` или `partial`;
- в evidence есть ссылка на релевантный пункт/узел;
- в debug trace видны шаги controller-а и выбранные `evidence_id`.

## 3. Вопрос без явной нормы

Шаги:
1. Ввести вопрос:
   `Какая минимальная высота ограждения кровли?`
2. Дождаться ответа.

Проверить:
- система не возвращает пустой экран и не зависает;
- система не требует обязательного указания СП до начала поиска;
- отображается ответ или частичный ответ;
- присутствует evidence panel.

Ожидаемый результат:
- режим ответа не должен быть пустым;
- допустим `partial`, если evidence ограничен;
- в evidence должны быть показаны найденные фрагменты или хотя бы один релевантный нормативный блок.

## 4. Проверка partial mode

Шаги:
1. Задать вопрос с неполным контекстом, например:
   `Нужен ли температурный шов?`

Проверить:
- система не делает слишком уверенный ответ без подтверждений;
- если evidence слабый, режим ответа переключается в `partial` или `clarify`;
- в блоке ограничений явно указано, что данных недостаточно.

Ожидаемый результат:
- при недостаточном evidence не должен появляться уверенный `direct`;
- должны быть показаны ограничения ответа.

## 5. Проверка clarify / no-answer поведения

Шаги:
1. Задать слишком общий или неинженерный вопрос, например:
   `Что лучше для строительства вообще?`
2. Задать заведомо вне корпуса вопрос, например:
   `Какие правила проектирования для марсианской базы?`

Проверить:
- система не выдумывает уверенный нормативный ответ;
- режим ответа становится `clarify` или `no_answer`;
- в ограничениях объяснено, почему ответа нет или что нужно уточнить.

Ожидаемый результат:
- нет hallucinated ответа;
- пользователь получает понятное ограничение или запрос на уточнение.

## 6. Проверка evidence panel

Для любого ответа проверить:
- каждый evidence item имеет `evidence_id`;
- у evidence отображаются:
  - локатор;
  - heading path;
  - source kind;
  - текст фрагмента;
- evidence соответствует теме вопроса;
- evidence не пустой, если answer mode `direct` или `partial`.

Ожидаемый результат:
- evidence panel читаема;
- можно глазами понять, откуда взят ответ;
- evidence не выглядит случайным или не по теме.

## 7. Проверка debug trace

Для любого ответа открыть `Debug Trace`.

Проверить:
- показан `policy hint`;
- показан список selected evidence ids;
- в trace видны observation/tool steps controller-а;
- trace соответствует фактическому вопросу.

Ожидаемый результат:
- debug trace помогает понять, что делал controller;
- trace не пустой;
- trace не противоречит evidence panel.

## 8. Проверка нескольких запросов в одной сессии

Шаги:
1. Последовательно задать 3-4 разных вопроса.
2. Убедиться, что история сообщений сохраняется на экране.

Проверить:
- предыдущие ответы не исчезают;
- новые ответы добавляются корректно;
- evidence/debug отображаются у соответствующего ответа;
- UI не ломается после нескольких сообщений.

Ожидаемый результат:
- чатовая сессия работает стабильно;
- визуально не путаются user/assistant сообщения.

## 9. Негативные признаки, считающиеся провалом ручной приемки

Сценарий считается проваленным, если наблюдается хотя бы одно из следующих:
- UI падает с traceback;
- ответ пустой при успешном завершении запроса;
- ответ уверенный, но evidence отсутствует;
- `direct` выдается без подтвержденных evidence;
- evidence явно не относится к вопросу;
- debug trace полностью пустой при включенном debug;
- приложение перестает отвечать после нескольких запросов.

## 10. Минимальный итог приемки

MVP можно считать вручную приемлемым, если:
- happy path с явной нормой работает;
- вопрос без явной нормы не ломает retrieval;
- partial/clarify режимы выглядят адекватно;
- evidence panel помогает проверить ответ;
- debug trace позволяет понять ход controller-а;
- UI стабилен хотя бы на короткой локальной сессии из нескольких вопросов.
