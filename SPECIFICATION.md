# Спецификация проекта

---


## Общая цель проекта

Система предназначена для:

- формирования и поддержания локальной базы нормативных документов;
- автоматической загрузки, нормализации и индексации документов из `meganorm.ru`;
- использования нормативной базы как основного слоя доказательной информации для агентной системы;
- инженерного консультирования пользователей по прикладным запросам, в том числе когда нормативный аспект не сформулирован явно;
- обязательного сопровождения каждого существенного тезиса ссылками на конкретные нормы, редакции, локаторы, цитаты и иные подтверждающие источники;
- параллельной проверки актуальности используемых документов без обязательной блокировки ответа;
- использования дополнительных доверенных и внешних источников как вспомогательных каналов с явной маркировкой их ненормативного статуса;
- предоставления системы через chat-based интерфейсы с изолированным сессионным контекстом;
- расширения агентной платформы новыми функциями, навыками, источниками, субагентами и моделями разных вендоров без переписывания ядра.

Реализация делится на два крупных этапа:

1. Этап 1: реализация парсера, загрузки, обработки и создания базы нормативных документов.
2. Этап 2: реализация расширяемой агентной системы инженерного консультирования с опорой на нормативную базу и дополнительные источники.

---

## Этап 1. Парсер, загрузка и создание базы нормативных документов

### 1.1. Цель этапа

На первом этапе должна быть создана локальная, структурированная и индексируемая база нормативных документов, пригодная для последующего использования агентной системой.

Результатом этапа должны стать:

- работающий парсер источников;
- хранилище исходных артефактов;
- хранилище метаданных и версий;
- структурированное представление документов;
- поисковые индексы для дальнейшего retrieval;
- механизм обновления устаревших документов.

### 1.2. Источники начальной загрузки

Начальная база должна формироваться только из следующих разделов:

1. `https://meganorm.ru/mega_doc/norm/sp_svod-pravil/sp_svod-pravil_0.html`
2. `https://meganorm.ru/mega_doc/norm/snip_stroitelnye-normy-i-pravila/snip_stroitelnye-normy-i-pravila_0.html`
3. `https://meganorm.ru/list2/64522-0.htm`
4. `https://meganorm.ru/list2/64511-0.htm`

Требования к обходу:

- парсер должен обходить все страницы пагинации в каждом разделе;
- парсер должен извлекать все ссылки на карточки документов;
- парсер должен повторно обходить карточку каждого документа для подтверждения статуса и извлечения метаданных.

### 1.3. Правила отбора документов

В базу должны включаться только документы с нормализованным статусом `active`.

К статусу `active` относятся:

- `действует`
- `действующий`
- `взамен`

К статусу `inactive` относятся:

- `утратил силу`
- `не действует`
- `заменен`
- `отменен`
- `частично отменен`
- `проект`
- `приостановлено действие`

Правила отбора:

- статус должен извлекаться из списка;
- статус должен повторно проверяться на карточке документа;
- при расхождении статусов приоритет у карточки документа;
- документы со статусом `unknown` не должны попадать в активную базу и должны логироваться для ручной проверки;
- правила нормализации статусов должны задаваться через конфиг, без жесткого зашивания в бизнес-логику.

### 1.4. Функциональные требования к парсеру

Парсер должен:

- обходить заданные разделы и их пагинацию;
- собирать карточки документов;
- определять способ представления документа;
- скачивать исходные артефакты;
- извлекать текст;
- извлекать и нормализовать метаданные;
- выполнять дедупликацию;
- ставить документы в очередь на структурную обработку и индексацию;
- поддерживать повторный запуск без повреждения уже загруженных данных.

### 1.5. Поддерживаемые форматы документов

Система должна поддерживать:

- HTML-документы с полным текстом;
- PDF-документы;
- документы, в которых контент представлен изображениями страниц;
- смешанный формат, где карточка содержит HTML-метаданные, а содержимое документа доступно в PDF.

### 1.6. Логика выбора источника контента

Приоритет извлечения контента:

1. Полный HTML-текст документа.
2. Печатная или альтернативная текстовая HTML-версия.
3. PDF-файл.
4. Изображения страниц как fallback.

Правила:

- HTML используется как основной источник для структурного разбора, если содержит полный текст;
- PDF сохраняется как канонический raw-артефакт, если доступен;
- если HTML неполный, приоритет переходит к PDF;
- если документ доступен только в виде сканов или изображений, должен применяться OCR.

### 1.7. Извлечение и хранение метаданных

Для каждого документа должны сохраняться, если доступны:

- наименование документа;
- обозначение;
- тип документа;
- исходный статус;
- нормализованный статус;
- дата актуализации текста;
- дата актуализации описания;
- дата издания;
- дата введения;
- сведения о редакции;
- область применения;
- нормативные ссылки;
- URL страницы списка;
- URL карточки;
- URL PDF;
- URL HTML-версии;
- URL печатной версии;
- дата загрузки;
- хэш содержимого;
- источник загрузки.

### 1.8. Извлечение текста и OCR

Система должна поддерживать два режима извлечения:

- прямое извлечение текста из HTML/PDF;
- OCR для сканов и изображений страниц.

Требования:

- текст должен извлекаться с сохранением логического порядка;
- при наличии страниц должна сохраняться привязка к страницам;
- OCR должен использоваться только как fallback;
- документ с низким качеством OCR должен помечаться как `low_confidence_parse`;
- сырой результат OCR должен сохраняться отдельно от нормализованного текста.

### 1.9. Структурная нормализация документа

После извлечения текст должен быть разбит на структурные узлы:

- документ;
- раздел;
- подраздел;
- пункт;
- подпункт;
- абзац;
- приложение;
- таблица;
- примечание.

Для каждого узла должны сохраняться:

- `node_id`
- `parent_node_id`
- `node_type`
- `label`
- `title`
- `text`
- `order_index`
- `page_from`
- `page_to`
- `char_start`
- `char_end`
- `parse_confidence`

### 1.10. Дедупликация и версионность

Система должна исключать дубликаты и хранить историю версий.

Правила дедупликации:

- основной ключ: нормализованное обозначение документа;
- дополнительный ключ: хэш содержимого;
- один документ может иметь несколько источников и несколько версий;
- в активном индексе должна участвовать только одна актуальная версия документа.

Для каждой версии должны храниться:

- идентификатор версии;
- дата обнаружения;
- редакция;
- статус;
- хэш;
- источник;
- признак активной версии;
- признак устаревшей версии.

### 1.11. Обновление устаревших документов

Система должна поддерживать автоматическое обновление документа.

Если обнаружено, что локальная версия устарела:

- должна быть скачана новая версия;
- должна быть выполнена полная повторная обработка;
- должна быть перестроена структура документа;
- должны быть обновлены поисковые индексы;
- новая версия должна стать активной только после успешного завершения всех этапов;
- старая версия должна оставаться в истории.

### 1.12. Хранилище данных

На первом этапе должно быть реализовано минимум четыре уровня хранения:

1. `Raw storage`
2. `Metadata storage`
3. `Structured storage`
4. `Search storage`

`Raw storage` должен содержать:

- исходный HTML;
- исходный PDF;
- изображения страниц;
- сырой OCR-вывод.

`Metadata storage` должен содержать:

- карточки документов;
- статусы;
- даты;
- ссылки;
- историю версий.

`Structured storage` должен содержать:

- нормализованное дерево документа;
- структурные узлы;
- ссылки между документами;
- локаторы.

`Search storage` должен содержать:

- полнотекстовый индекс;
- векторный индекс;
- индекс по метаданным.

Минимальные сущности:

- `documents`
- `document_versions`
- `document_sources`
- `document_nodes`
- `document_references`
- `ingestion_jobs`
- `update_events`

### 1.13. Индексация и подготовка к RAG

Поиск должен быть гибридным и опираться на структурное представление документа, но production retrieval не должен выполняться напрямую по каждому структурному узлу.

Обязательные виды индексации:

- полнотекстовый поиск;
- поиск по метаданным;
- векторный поиск;
- фильтрация по типу документа, статусу, дате редакции и источнику.

Архитектурные требования:

- `document_nodes` являются каноническим структурным слоем и хранят иерархию документа, локаторы и точные фрагменты текста;
- поверх `document_nodes` должен существовать отдельный search-оптимизированный слой `retrieval_chunks`;
- единицей retrieval на уровне базы должен быть не отдельный структурный узел, а оптимизированный retrieval-chunk, состоящий из одного или нескольких связанных структурных узлов;
- embeddings не должны храниться на каждом `document_node`, если это приводит к существенному раздуванию базы;
- embeddings должны строиться только для search-оптимизированного слоя и только там, где это оправдано для retrieval.

Базовые правила chunking:

- chunk должен строиться из логически связанного нормативного блока, а не из одной строки;
- типичный chunk должен соответствовать пункту с дочерними подпунктами, короткому подразделу или компактному смысловому блоку;
- слишком короткие фрагменты допускается объединять с соседними связанными узлами;
- слишком длинные фрагменты должны разбиваться на несколько chunks без избыточного overlap;
- канонические локаторы и цитаты при этом должны восстанавливаться через исходные `document_nodes`.

### 1.14. Устойчивость и эксплуатационные требования

Парсер должен:

- соблюдать rate limiting;
- поддерживать таймауты;
- поддерживать повторные попытки при сетевых ошибках;
- логировать ошибки и нестандартные статусы;
- поддерживать идемпотентный повторный запуск;
- поддерживать продолжение обработки после прерывания;
- обеспечивать возможность переиндексации без повторного скачивания raw-данных;
- обеспечивать возможность повторного OCR без повторного скачивания raw-данных.

### 1.15. Результат этапа 1

Этап 1 считается завершенным, если:

- все 4 исходных раздела обходятся полностью;
- в базу попадают только документы со статусом `active`;
- документы со статусом `взамен` включаются в базу;
- документы с неактивными статусами не попадают в активный индекс;
- для каждого загруженного документа сохранены raw-артефакты и метаданные;
- для каждого документа построено структурированное представление;
- создан гибридный индекс для поиска;
- реализован механизм обновления устаревших документов.

## Этап 2. Агентная система инженерного консультирования

### 2.1. Цель этапа

На втором этапе должна быть реализована расширяемая агентная система, которая использует базу нормативных документов, созданную на первом этапе, как основной слой доказательной базы, но не ограничивается только прямыми вопросами по нормам.

Система должна выступать как полезный консультант инженера-проектировщика: анализировать запрос, самостоятельно выявлять нормативный аспект, декомпозировать задачу, извлекать подтверждения из локальной нормативной базы, при необходимости использовать дополнительные источники и формировать ответ с явным разграничением уровня доверия к каждому тезису.

Результатом этапа должны стать:

- основной агент-оркестратор;
- расширяемый набор специализированных субагентов и инструментов;
- контур анализа, декомпозиции и маршрутизации пользовательского запроса;
- retrieval по базе нормативных документов через оптимизированный chunk-layer с привязкой к структурным узлам;
- неблокирующая проверка актуальности используемых документов;
- fallback на доверенные внешние источники и общий web-поиск;
- финальный контур верификации полноты, ссылочности, безопасности и достоверности ответа;
- единое прикладное ядро, доступное через web UI и Telegram.

### 2.2. Общие требования к агентной системе

Система должна:

- принимать запрос пользователя в свободной форме;
- уметь работать не только с прямыми вопросами по нормам, но и с прикладными инженерными запросами, в которых нормативный аспект не указан явно;
- самостоятельно определять, требуется ли обращение к нормативной базе;
- анализировать запрос, выделять его смысловые аспекты и при необходимости разбивать его на подзадачи;
- использовать нормативную базу как приоритетный источник доказательств;
- при недостаточности нормативных данных уметь использовать дополнительные внешние источники;
- формировать ответ только на основе подтвержденных evidence-блоков с явным указанием происхождения каждого тезиса;
- удерживать контекст только в пределах текущей пользовательской сессии;
- поддерживать несколько независимых пользовательских сессий одновременно;
- обеспечивать изоляцию контекста, данных и промежуточных артефактов между сессиями.
- обеспечивать изоляцию истории сообщений, состояния инструментов, фоновых задач, временных файлов и промежуточных результатов между сессиями.

### 2.3. Базовая архитектура

Для этапа 2 должна использоваться архитектура orchestrator-first.

Архитектурные принципы:

- один основной агент-оркестратор;
- набор специализированных субагентов или модулей, вызываемых оркестратором по задаче;
- единое прикладное ядро, не зависящее от конкретного канала доступа;
- единый реестр инструментов и источников;
- явные адаптеры к моделям, embedding-провайдерам, поиску и внешним каналам;
- проектирование с запасом на последующее расширение состава субагентов, навыков и источников данных без переписывания ядра.

Минимальный состав модулей этапа 2:

- Orchestrator
- Query Analyzer
- Task Decomposer
- Normative Retriever
- Freshness Checker
- Web Researcher
- Answer Synthesizer
- Citation Auditor
- Coverage Auditor
- Hallucination Guard
- Safety Guard

Архитектура должна допускать дальнейшее добавление новых специализированных субагентов, в том числе предметных, расчетных, сравнительных и ревизионных.

### 2.4. Назначение модулей

Orchestrator отвечает за:

- управление полным жизненным циклом запроса;
- маршрутизацию запроса между субагентами и инструментами;
- параллельный запуск retrieval, freshness-check и вспомогательных задач;
- сбор промежуточных результатов и формирование финального ответа;
- контроль таймаутов, деградационных сценариев и итогового статуса ответа.

Query Analyzer отвечает за:

- нормализацию вопроса;
- определение типа запроса;
- выделение темы, сущностей и ограничений;
- выявление нормативных и ненормативных аспектов;
- определение, нужны ли дополнительные источники кроме нормативной базы.

Task Decomposer отвечает за:

- разбиение сложного вопроса на подзадачи;
- выделение аспектов, которые должны быть закрыты в ответе;
- формирование плана вызова субагентов и инструментов.

Normative Retriever отвечает за:

- поиск релевантных документов, версий и retrieval-chunks;
- гибридный retrieval по FTS, метаданным и векторному поиску;
- ранжирование evidence-кандидатов;
- возврат chunk-кандидатов с последующим восстановлением точных локаторов и цитат через `document_nodes`;
- возврат нормализованных evidence-объектов для дальнейшего анализа.

Freshness Checker отвечает за:

- проверку актуальности документа в фоне во время подготовки ответа;
- сравнение локальной версии с источником;
- запуск обновления при необходимости;
- возврат статуса свежести без блокировки ответа;
- передачу в ответ информации о том, какая локальная версия использована и есть ли более актуальная версия в источнике.

Web Researcher отвечает за:

- поиск вспомогательной информации вне нормативной базы;
- приоритетный поиск по списку особенно доверенных сайтов;
- fallback на общий web-поиск;
- возврат только явно размеченных ненормативных evidence-объектов с указанием источника.

Answer Synthesizer отвечает за:

- сбор evidence из нескольких каналов;
- формирование ответа с явным разделением нормативных и ненормативных тезисов;
- отделение прямой цитаты нормы от аналитического вывода;
- включение предупреждений о неполноте, устаревании версии или необходимости ручной проверки.

Citation Auditor отвечает за:

- проверку, что каждый существенный тезис снабжен ссылкой;
- проверку наличия конкретного локатора;
- проверку наличия цитаты или точного фрагмента опоры;
- запрет тезисов без доказательной базы.

Coverage Auditor отвечает за:

- проверку, что все важные аспекты вопроса закрыты;
- выявление пробелов и неотвеченных подпунктов;
- маркировку ответа как неполного, если покрытие недостаточно.

Hallucination Guard отвечает за:

- выявление утверждений, не подтвержденных источниками;
- запрет вымышленных фактов и несанкционированных обобщений;
- перевод ответа в режим отказа, предупреждения или ограничения, если evidence недостаточно.

Safety Guard отвечает за:

- защиту от prompt injection;
- защиту от выполнения скрытых инструкций из retrieved-контента;
- контроль политик доступа к инструментам;
- фильтрацию внешнего контента перед его включением в LLM-контекст;
- аудит вызовов инструментов и событий безопасности.

### 2.5. Пайплайн обработки запроса

Обработка запроса должна включать следующие шаги:

1. Прием запроса пользователя в канале общения.
2. Нормализация запроса и определение его типа.
3. Выделение аспектов, подзадач и целевых источников.
4. Запуск retrieval по нормативной базе.
5. Параллельный запуск проверки актуальности документов, выбранных для ответа.
6. При необходимости запуск trusted web search и общего web-search как вспомогательных каналов.
7. Сбор evidence из нормативной базы и дополнительных источников.
8. Формирование черновика ответа.
9. Проверка ссылок, локаторов, цитат и происхождения каждого тезиса.
10. Проверка полноты покрытия вопроса.
11. Проверка на неподтвержденные утверждения и нарушения политик безопасности.
12. Возврат финального ответа пользователю.

Если обновление документа уже инициировано, но еще не завершено, ответ не должен блокироваться. Система может использовать локальную версию, но обязана явно пометить, что использована не самая актуальная редакция, если это установлено проверкой актуальности.

### 2.6. Требования к retrieval и источникам информации

Агентная система должна использовать многоуровневую схему источников.

Приоритет источников:

1. Локальная нормативная база.
2. Особенно доверенные внешние сайты.
3. Общий web-поиск.

Требования к retrieval:

- нормативный поиск должен выполняться на уровне search-оптимизированных retrieval-chunks, а не напрямую по каждому `document_node`;
- поиск должен поддерживать полнотекстовый, семантический и метаданный режимы;
- результаты должны ранжироваться по релевантности, качеству извлечения и качеству источника;
- документы со статусом, отличным от active, не должны использоваться как нормативная опора;
- узлы с низким parse_confidence не должны использоваться как единственная опора для ответа;
- trusted web search должен использовать allowlist доверенных доменов;
- общий web-поиск должен использоваться только как вспомогательный, а не как замена нормативной базы.

Требования к архитектуре retrieval layer:

- `document_nodes` должны оставаться каноническим структурным слоем для локаторов, иерархии и точного цитирования;
- поверх `document_nodes` должен строиться отдельный слой `retrieval_chunks`, оптимизированный для FTS и vector retrieval;
- embeddings должны создаваться только для `retrieval_chunks`, а не для каждого структурного узла;
- embeddings должны строиться только для активных версий документов;
- идентичные по тексту chunks не должны переэмбеддиться повторно и должны дедуплицироваться по стабильному `chunk_hash`;
- embeddings должны храниться отдельно от канонического структурного слоя, чтобы смена модели или размерности не требовала переписывания `document_nodes`;
- vector index должен строиться только для актуального набора active chunks и активного embedding set;
- exact-match, code lookup и locator-oriented запросы должны по возможности обслуживаться без обязательного запуска dense retrieval;
- dense retrieval должен использоваться тогда, когда он действительно дает выигрыш по качеству поиска, а не как обязательный шаг для любого запроса.

Требования к rollout dense retrieval:

- целевое dense-покрытие должно строиться для всего active нормативного корпуса, а не для отдельного подмножества типов документов;
- отсутствие embeddings для части корпуса на промежуточном этапе построения не должно ломать retrieval: система должна деградировать контролируемо и использовать доступные режимы поиска;
- до завершения полного backfill допустим временный fallback на `FTS + metadata + exact-match` для документов, embeddings для которых еще не построены;
- массовая генерация embeddings для всего active корпуса должна запускаться только после предварительной оценки качества, стоимости и ожидаемого объема хранения.

Требования к оценке стоимости embeddings:

- перед массовым созданием embeddings для retrieval-chunks должна выполняться предварительная оценка объема корпуса;
- оценка должна включать как минимум количество active документов корпуса, ожидаемое количество retrieval-chunks, примерный объем текста или токенов, оценку дискового размера `chunk_embeddings` и vector index, а также ориентировочную стоимость генерации embeddings выбранной моделью;
- без такой оценки массовая генерация embeddings для корпуса не должна считаться готовой к запуску в production-like режиме;
- pipeline должен поддерживать dry-run режим оценки без фактической записи embeddings в базу.

Требования к chunking:

- chunk должен строиться из логически завершенного нормативного блока;
- предпочтительная единица chunking: пункт вместе с дочерними подпунктами и короткими абзацами;
- заголовки разделов и подразделов могут использоваться как контекст chunk-а, но не должны автоматически становиться самостоятельными embedding-единицами;
- слишком короткие chunks допускается объединять с соседними sibling-блоками того же parent-контекста;
- слишком длинные chunks должны разрезаться на несколько search-оптимизированных частей;
- необходимо минимизировать избыточный overlap между chunks, чтобы не раздувать БД и не повышать стоимость embeddings без реального прироста качества;
- chunking должен быть детерминированным, чтобы при повторной индексации можно было стабильно переиспользовать `chunk_hash` и embeddings.

Каждый найденный фрагмент нормативного retrieval должен возвращаться с:

- идентификатором документа;
- идентификатором версии;
- идентификатором retrieval-chunk;
- названием документа;
- обозначением документа;
- статусом;
- идентификатором начального узла;
- идентификатором конечного узла;
- локатором узла или диапазоном локаторов;
- текстом фрагмента;
- страницей, если доступна;
- URL источника;
- score;
- parse_confidence.

Каждый внешний evidence-объект должен дополнительно содержать:

- тип внешнего источника (	rusted_web или open_web);
- домен источника;
- признак ненормативного происхождения;
- предупреждение о необходимости проверки.

### 2.7. Неблокирующая проверка актуальности

Проверка актуальности должна выполняться параллельно с подготовкой ответа и не должна блокировать выдачу ответа пользователю.

Проверка актуальности должна запускаться, если:

- документ давно не проверялся;
- есть признаки устаревшей версии;
- документ участвует в критичном ответе;
- пользовательский запрос требует максимально актуальной редакции.

Логика:

- сравнение локальных метаданных с источником;
- сравнение дат актуализации;
- сравнение статуса и редакции;
- при необходимости запуск обновления;
- возврат статуса `fresh`, `stale`, `refresh_in_progress`, `refresh_failed` или аналогичного нормализованного результата.

Если проверка показала, что локальная версия устарела, но обновление еще не завершено, система может использовать локальную версию в ответе только при выполнении всех условий:

- пользователь явно информируется, что использована локальная версия;
- указывается, какая редакция использована локально;
- указывается, что в источнике обнаружена более актуальная редакция;
- указывается, что итог требует повторной проверки после завершения обновления.

Если обновление завершилось неуспешно, система не должна представлять такой документ как гарантированно актуальный источник. Ответ должен быть ограничен и снабжен явным предупреждением.

### 2.8. Требования к формированию ответа

Ответ должен формироваться только на основе подтвержденных evidence-блоков, при этом ответ может содержать несколько слоев достоверности.

Каждый финальный ответ должен содержать:

- прямой ответ на запрос пользователя;
- разделение между нормативными утверждениями и ненормативными вспомогательными сведениями;
- указание условий, допущений и ограничений;
- ссылки на конкретные нормы, если нормативная опора найдена;
- указание редакции документа;
- указание раздела, пункта или подпункта;
- короткие релевантные цитаты;
- список использованных источников;
- пометки о свежести версии, если версия не подтверждена как актуальная;
- предупреждение, если часть ответа основана на web-поиске или ином ненормативном источнике.

Запрещено:

- ссылаться только на документ в целом без конкретного локатора, если локатор доступен;
- включать существенные тезисы без ссылки;
- выдавать неподтвержденные нормативные утверждения;
- выдавать информацию из web-поиска как нормативную;
- смешивать цитату нормы и интерпретацию без явного разграничения;
- скрывать от пользователя, что использована потенциально устаревшая версия.

### 2.9. Формат evidence-объекта

Для внутренней работы системы должен использоваться стандартизированный evidence-объект.

Минимальная структура:

```json
{
  "source_kind": "normative|trusted_web|open_web",
  "document_id": "string|null",
  "document_version_id": "string|null",
  "chunk_id": "string|null",
  "document_title": "string",
  "document_code": "string|null",
  "status": "active|inactive|unknown|null",
  "edition_label": "string|null",
  "node_id": "string|null",
  "start_node_id": "string|null",
  "end_node_id": "string|null",
  "locator": "п. 4.2, абз. 2",
  "locator_end": "п. 4.2.3",
  "page": 12,
  "chunk_text": "string|null",
  "quote": "string",
  "quote_char_range": [100, 180],
  "source_url": "string",
  "source_domain": "string",
  "relevance_score": 0.92,
  "parse_confidence": 0.97,
  "freshness_status": "fresh|stale|refresh_in_progress|refresh_failed|unknown",
  "is_normative": true,
  "requires_verification": false
}
```

Для нормативного retrieval evidence-объект должен одновременно нести два уровня представления:

- retrieval-level: `chunk_id`, `chunk_text`, агрегированный score и границы chunk-а;
- citation-level: `node_id` либо `start_node_id/end_node_id`, точный locator и цитату, по которым можно построить доказательный ответ.

### 2.10. Контур верификации

Перед выдачей ответа пользователю система обязана выполнить как минимум следующие проверки:

1. Проверка полноты покрытия вопроса.
2. Проверка доказательной базы.
3. Проверка на ложные или вымышленные утверждения.
4. Проверка корректной маркировки происхождения источников.
5. Проверка предупреждений о свежести и необходимости ручной проверки.

Требования:

- если не закрыты все ключевые аспекты вопроса, ответ должен быть помечен как неполный;
- если у тезиса нет конкретной ссылки, он должен быть удален из ответа или помечен как непроверенный;
- если система не может подтвердить нормативный вывод, она должна явно указать на недостаточность подтверждений;
- если тезис опирается на внешний web-поиск, это должно быть явно указано;
- при низком качестве OCR, слабом evidence или stale-версии ответ должен содержать предупреждение о необходимости ручной проверки.

### 2.11. Память, сессии и каналы доступа

На этапе 2 память системы должна быть только сессионной.

Требования:

- новая сессия или новый чат создают новый изолированный контекст;
- долгосрочная пользовательская память не требуется и не должна быть обязательной частью этапа 2;
- допускается хранение краткого summary контекста только в пределах сессии;
- система должна поддерживать несколько одновременных пользовательских сессий;
- контексты разных пользователей и разных сессий должны быть изолированы;
- изоляция должна распространяться на историю сообщений, stateful tool-context, очереди фоновых задач, кэш, временные артефакты и внутренние промежуточные данные;
- должен существовать web UI в формате чата;
- должен существовать адаптер доступа через Telegram-бота;
- web UI и Telegram должны использовать одно и то же прикладное ядро и одинаковые правила оркестрации.

### 2.12. Асинхронность и исполняющая среда

Система должна работать асинхронно и поддерживать одновременную работу нескольких пользователей.

Требования:

- несколько сессий должны обрабатываться параллельно;
- фоновые задачи проверки актуальности, refresh и web-research не должны блокировать основные пользовательские запросы;
- должна быть предусмотрена возможность стриминга промежуточного и финального ответа в UI;
- состояние выполнения агентного пайплайна должно быть отслеживаемым;
- архитектура должна допускать последующее введение более автономных субагентов без полной переработки ядра.

### 2.13. Безопасность

Для этапа 2 должна быть предусмотрена отдельная система безопасности.

Обязательные требования:

- защита от prompt injection из пользовательского ввода, документов и web-источников;
- retrieved-контент не должен интерпретироваться как системная инструкция;
- внешние источники должны проходить санитарную обработку перед включением в prompt;
- вызовы инструментов должны выполняться только через контролируемый слой доступа;
- должен существовать allowlist доверенных доменов для trusted web search;
- должен существовать audit trail по ключевым событиям: запрос, retrieval, tool calls, refresh, финальный ответ;
- должны применяться лимиты и изоляция на уровне пользовательских сессий;
- механизмы изоляции должны предотвращать утечки между сессиями через shared cache, background workers, tool state и временные артефакты;
- архитектура должна позволять безопасное подключение новых инструментов и субагентов.

### 2.14. Абстракция моделей и расширяемость

Архитектура этапа 2 не должна зависеть от одного поставщика моделей.

Требования:

- должен существовать абстрактный слой для ChatModelProvider;
- должен существовать абстрактный слой для EmbeddingProvider;
- при необходимости должны быть предусмотрены отдельные абстракции для RerankerProvider, OCRProvider и других модельных сервисов;
- подключение новых моделей, в том числе от других вендоров и локальных open-source моделей, должно выполняться без переписывания бизнес-логики оркестратора;
- выбор конкретной модели должен быть конфигурируемым;
- система должна поддерживать разные модели для разных ролей: orchestration, synthesis, embeddings, reranking и последующих специализированных задач.

### 2.15. Минимальные сущности этапа 2

На втором этапе должны быть реализованы как минимум следующие сущности хранения:

- retrieval_chunks
- chunk_embeddings
- qa_sessions
- qa_queries
- qa_subtasks
- qa_evidence
- qa_answers
- verification_reports
- tool_invocations
- freshness_checks
- security_events

### 2.16. Ограничения этапа 2

На этапе 2 система ориентирована на инженерное консультирование с опорой на нормативную базу и не должна позиционироваться как юридическое заключение или как полностью автономный эксперт без необходимости профессиональной верификации в сложных случаях.

Допущения:

- интерпретация нормы допустима только при наличии прямых подтверждающих цитат;
- если данные противоречивы или недостаточны, система должна явно сообщать об этом;
- внешний web-поиск может использоваться как вспомогательный канал, но не как замена нормативному источнику;
- ненормативные источники должны быть явно помечены как требующие проверки;
- долгосрочная межсессионная память не входит в объем этапа 2.

### 2.17. Результат этапа 2

Этап 2 считается завершенным, если:

- система принимает и обрабатывает запросы пользователя в web UI;
- система доступна через Telegram-бота;
- сложные запросы разбиваются на подзадачи;
- оркестратор использует специализированные модули или субагенты;
- нормативный retrieval выполняется по оптимизированным retrieval-chunks и возвращает конкретные нормы на уровне пунктов и подпунктов;
- embeddings создаются только для active retrieval-chunks и не дублируются без необходимости для идентичных текстовых блоков;
- dense retrieval после завершения backfill должен покрывать весь active нормативный корпус, при этом точные `code lookup` и `locator-oriented` запросы по-прежнему могут обслуживаться через `FTS + metadata + exact-match`, когда это эффективнее;
- перед массовой генерацией embeddings для retrieval-chunks выполнена оценка количества chunks, примерного объема токенов, ожидаемого размера хранения и ориентировочной стоимости выбранной embedding-модели;
- проверка актуальности выполняется параллельно с подготовкой ответа;
- при обнаружении устаревшей версии система инициирует обновление, но не блокирует ответ без необходимости;
- ответ явно разграничивает нормативные и ненормативные источники;
- финальный ответ не содержит существенных тезисов без ссылок;
- каждый существенный вывод снабжен документом, редакцией, локатором и цитатой либо явной пометкой о внешнем происхождении;
- система выявляет неполное покрытие вопроса;
- система защищена от prompt injection на уровне архитектурных правил и контуров безопасности;
- архитектура допускает подключение новых моделей, новых источников и новых субагентов без переписывания ядра.
## Общие критерии приемки проекта

Проект считается реализованным в рамках MVP, если:

- этап 1 полностью формирует пригодную для поиска и версионирования базу документов;
- этап 2 использует эту базу для доказательных ответов;
- все ответы строятся только на подтвержденных данных;
- все документы в активной базе имеют нормализованный статус `active`;
- система умеет обновлять устаревшие версии документов;
- при недостатке подтверждений система не выдает категоричных или вымышленных утверждений.
