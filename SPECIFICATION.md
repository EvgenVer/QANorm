# Спецификация проекта

## Общая цель

QANorm на текущем этапе предназначен только для формирования и поддержания локальной базы нормативных документов.

Система должна:

- обходить заданные seed-разделы;
- извлекать карточки и метаданные документов;
- загружать raw-артефакты;
- извлекать текст из HTML/PDF и применять OCR при необходимости;
- нормализовать документ до структурных узлов;
- хранить документы, версии, источники, raw-файлы и историю обновлений;
- обеспечивать локальную индексацию и переиндексацию Stage 1.

## Источники

Начальная загрузка должна выполняться только из seed URL, определенных в `configs/sources.yaml`.

Требования:

- обход пагинации обязателен;
- карточка документа должна перепроверяться отдельно;
- статус документа должен нормализоваться через конфиг, а не через жестко зашитые строки.

## Правила отбора документов

В локальную активную базу включаются только документы со статусом `active`.

Требования:

- статус из списка и карточки должен сверяться;
- при конфликте приоритет имеет карточка документа;
- документы со статусом `unknown` не попадают в активную базу и должны быть доступны для последующего анализа.

## Поддерживаемые форматы

Система должна поддерживать:

- HTML с полным текстом;
- PDF;
- изображения страниц;
- смешанный карточка + PDF сценарий.

Приоритет извлечения:

1. Полный HTML.
2. Альтернативный HTML.
3. PDF.
4. OCR по изображениям страниц.

## Хранение и модель данных

Система должна хранить:

- канонический документ;
- версии документа;
- источники обнаружения;
- raw-артефакты;
- структурные узлы документа;
- ссылки между документами;
- очередь ingestion-задач;
- события обновления.

База должна поддерживать:

- дедупликацию по нормализованному обозначению и содержимому;
- одну активную версию документа;
- сохранение исторических версий;
- повторяемый reindex без повреждения данных.

## Обработка текста

Требования:

- текст должен извлекаться с сохранением логического порядка;
- OCR используется только как fallback;
- результат OCR должен храниться отдельно от нормализованного текста;
- низкое качество OCR должно отражаться в `parse_confidence` и метриках Stage 1.

## Нормализация структуры

После извлечения документ должен разбиваться на структурные узлы, включая:

- раздел;
- подраздел;
- пункт;
- подпункт;
- абзац;
- приложение;
- таблицу;
- примечание.

Для каждого узла должны сохраняться:

- тип;
- метка и заголовок;
- текст;
- порядок;
- границы по страницам и символам;
- `parse_confidence`.

## Индексация

Stage 1 должен поддерживать локальную node-level индексацию:

- полнотекстовый индекс по `document_nodes`;
- векторное поле на уровне `document_nodes` для локальных поисковых экспериментов;
- очистку и переиндексацию inactive/устаревших версий.

## Фоновые задачи

Очередь ingestion-задач должна жить в PostgreSQL.

Требования:

- worker атомарно захватывает задачу;
- временные ошибки должны переводить задачу в retry;
- постоянные ошибки должны фиксироваться в `last_error`;
- повторный запуск worker не должен ломать уже обработанные документы.

## Интерфейс управления

CLI должен предоставлять как минимум команды:

- `check-config`
- `health-check`
- `init-db`
- `crawl-seeds`
- `run-worker`
- `reindex`
- `refresh-document`
- `update-document`
- `ingestion-metrics`
- `ingestion-report`

## Исключения из текущей спецификации

В текущую спецификацию не входят:

- агентная система;
- консультативный retrieval/runtime слой;
- API;
- пользовательские интерфейсы консультативного слоя;
- Telegram;
- trusted/open web search;
- answer synthesis;
- verification/audit/security runtime слой.
