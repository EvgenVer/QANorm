# План реализации проекта

## Назначение документа

Документ фиксирует практический план реализации Этапа 1 и Этапа 2 проекта:

- принятый стек;
- ключевые архитектурные решения;
- архитектуру проекта;
- очередность разработки;
- модель данных;
- метрики качества;
- критерии готовности этапов.

Этап 1 охватывает парсер, загрузку, обработку, хранение и индексацию нормативных документов.

Этап 2 охватывает агентную систему инженерного консультирования поверх подготовленной нормативной базы, web UI, Telegram-канал, retrieval, проверку актуальности, web-research, безопасность, наблюдаемость и аудит.

---

## 1. Цель Этапа 1

Цель Этапа 1: построить устойчивый ingestion-пайплайн, который:

- обходит заданные разделы `meganorm.ru`;
- отбирает только документы со статусом `active` (включая `взамен`);
- загружает исходные артефакты документов;
- извлекает текст из HTML, PDF и сканов;
- нормализует документы до структурных узлов;
- сохраняет raw-данные, метаданные, версии и структуру;
- строит поисковые индексы для последующего RAG;
- обновляет устаревшие версии документов.

Результат этапа: локальная база нормативных документов, пригодная для retrieval на уровне пунктов и подпунктов.

## 2. Принятый стек

### 2.1. Язык и среда

- Язык разработки: `Python 3.12`
- Менеджер окружения и зависимостей: `uv`
- Формат конфигурации: `YAML`
- Управление настройками: `pydantic-settings`

### 2.2. Сетевой слой и устойчивость

- HTTP-клиент: `httpx`
- Повторные попытки и backoff: `tenacity`
- Rate limiting: собственная обертка в коде

### 2.3. Парсинг и обработка документов

- HTML-парсинг: `selectolax`
- Дополнительный парсинг сложных HTML: `lxml`
- PDF-обработка: `PyMuPDF`
- OCR-обвязка: `pytesseract`
- OCR-движок: `Tesseract OCR` с языками `rus` и `eng`

### 2.4. Хранение и индексация

- Основная БД: `PostgreSQL 16`
- Векторный поиск: `pgvector`
- Полнотекстовый поиск: `PostgreSQL Full Text Search`
- ORM / доступ к БД: `SQLAlchemy 2.x`
- Миграции: `Alembic`

### 2.5. Логирование и тестирование

- Логирование: стандартный `logging` с возможностью структурированного расширения
- Тестирование: `pytest`

### 2.6. Хранение raw-данных

Принятое решение:

- основное хранилище raw-артефактов: локальная папка на диске;
- эта папка может синхронизироваться с `Google Drive` внешним инструментом, но для проекта остается обычным локальным файловым хранилищем;
- БД хранит только метаданные, пути, хэши, статусы и структуру, но не сами крупные бинарные объекты.

Практическая схема:

- все raw-артефакты сохраняются в локальной иерархии каталогов;
- обработка, OCR, PDF-рендеринг и переобработка работают напрямую с локальными файлами;
- если папка синхронизируется с `Google Drive`, это считается внешней функцией среды, а не частью приложения;
- в БД сохраняются локический путь хранения и контрольный хэш файла.

Ограничение:

- приложение не должно зависеть от `Google Drive API`;
- приложение не должно содержать отдельный backend или клиент для работы с `Google Drive`;
- синхронизация с облаком не должна влиять на внутреннюю логику ingestion.

## 3. Основные архитектурные решения

### 3.1. Архитектурный стиль

Для Этапа 1 принимается модульный монолит:

- один репозиторий;
- один кодовый проект;
- отдельные логические модули для crawler, parser, storage, OCR и indexing;
- фоновая обработка через worker-процессы, без отдельного брокера сообщений.

Причины выбора:

- минимальная операционная сложность;
- быстрый запуск MVP;
- простое локальное разворачивание;
- возможность позже выделить части в сервисы без переписывания доменной логики.

### 3.2. Очереди и фоновые задачи

Очереди на MVP реализуются через таблицу задач в `PostgreSQL`.

Принятое решение:

- не использовать `RabbitMQ`, `Redis` или `Celery` на старте;
- использовать таблицу `ingestion_jobs` как источник задач;
- отдельный worker-процесс забирает задачи из БД и выполняет их.

Преимущества:

- меньше инфраструктуры;
- прозрачное логирование и отслеживание статусов;
- простая идемпотентность и восстановление после сбоев.

### 3.3. Идемпотентность

Все ключевые операции должны быть идемпотентными:

- повторный обход страницы не должен создавать дубли;
- повторная загрузка документа с тем же хэшем не должна порождать новую активную версию;
- повторная индексация не должна ломать структуру;
- повторный запуск обработки после сбоя должен продолжаться с сохраненного состояния.

### 3.4. Подход к версионности

Документ и версия документа должны быть разнесены.

Принятое решение:

- сущность `document` отражает канонический документ;
- сущность `document_version` отражает конкретную редакцию или обнаруженную версию;
- активной для поиска и последующего RAG является только одна версия документа;
- старые версии не удаляются и остаются в истории.

### 3.5. Подход к извлечению контента

Принцип обработки:

1. Если доступен полный HTML-текст, он используется как основной источник структуры.
2. Если HTML неполный, используется PDF.
3. Если PDF текстовый, извлекается текстовый слой.
4. Если PDF сканированный или страница состоит из изображений, запускается OCR.

Принятое решение:

- `HTML` является предпочтительным источником для структуры;
- `PDF` является каноническим raw-артефактом, если он доступен;
- `OCR` является fallback-режимом.

### 3.6. Подход к поиску

Для Этапа 1 принимается гибридный индекс:

- полнотекстовый индекс в `PostgreSQL`;
- векторный индекс в `pgvector`;
- отдельные фильтруемые поля по метаданным и статусу.

Единица поиска:

- не целый документ;
- структурный узел документа: пункт, подпункт, абзац, приложение, таблица.

Правила структуры:

- основной Python-пакет проекта: `src/qanorm`;
- вся бизнес-логика должна располагаться внутри `src/qanorm`;
- `scripts/` используются только как тонкие entrypoint-обертки;
- `data/` является локальным файловым хранилищем и не должна попадать в git;
- конфиги выносятся в `configs/`, чтобы статусы, сиды и лимиты менялись без правки кода;
- миграции БД размещаются в `alembic/`, а ORM-модели в `src/qanorm/models/`.

`src/qanorm/parsers`

- парсинг списков;

- парсинг HTML-текста;
- извлечение текста из PDF;

- рендеринг PDF-страниц в изображения;

- небольшие переиспользуемые функции без доменной логики.

### 4.3. Контуры данных

Контур обработки данных:

1. `Crawler` находит ссылки на документы.
2. `Jobs` создают задачи на обработку.
3. `Fetchers` загружают raw-артефакты.
4. `Parsers` извлекают метаданные и текст.
5. `OCR` обрабатывает сканы при необходимости.
6. `Normalizers` строят структуру и локаторы.
7. `Storage` сохраняет raw-артефакты и связанные ссылки.
8. `Indexing` обновляет полнотекстовый и векторный индексы.

## 5. Очередность разработки

### 5.1. Этапность работ

Разработка должна идти поэтапно, от базовой устойчивой загрузки к полной обработке.

### 5.2. Порядок реализации

#### Шаг 1. Базовая инфраструктура проекта

Реализовать:

- каркас проекта;
- управление конфигами;
- подключение к БД;
- миграции;
- базовое логирование;
- CLI для служебных команд.

Результат:

- проект запускается;
- миграции применяются;
- доступна базовая команда health-check / init.

#### Шаг 2. Базовая схема данных

Реализовать:

- таблицы документов;
- таблицы версий;
- таблицы источников;
- таблицы задач;
- таблицы raw-артефактов;
- таблицы структурных узлов;
- таблицы ссылок между документами.

Результат:

- можно сохранять карточку документа, версию, задачу и узлы.

#### Шаг 3. Crawler разделов и пагинации

Реализовать:

- обход 4 сидов;
- парсинг страниц списков;
- извлечение ссылок на карточки;
- извлечение статусов на уровне списка;
- создание задач на обработку документов.

Результат:

- система собирает список кандидатов на загрузку.

#### Шаг 4. Парсинг карточки документа и фильтрация статусов

Реализовать:

- извлечение метаданных карточки;
- повторную проверку статуса;
- нормализацию статусов;
- отбраковку `inactive` и `unknown`;
- нормализацию обозначения документа.

Результат:

- в очередь полной обработки попадают только `active` документы.

#### Шаг 5. Загрузка raw-артефактов

Реализовать:

- скачивание HTML;
- скачивание PDF;
- скачивание изображений страниц при необходимости;
- сохранение в локальное raw-хранилище;
- расчет и сохранение хэшей.

Результат:

- raw-артефакты доступны в хранилище и привязаны к версии документа.

#### Шаг 6. Извлечение текста

Реализовать:

- извлечение полного HTML-текста;
- извлечение текстового слоя из PDF;
- определение необходимости OCR;
- извлечение сырых текстовых блоков.

Результат:

- для версии документа есть текстовый материал для последующей нормализации.

#### Шаг 7. OCR fallback

Реализовать:

- рендеринг страниц;
- OCR;
- оценку качества OCR;
- маркировку `low_confidence_parse`.

Результат:

- даже сканированные документы проходят в базу с контролем доверия к качеству.

#### Шаг 8. Нормализация структуры документа

Реализовать:

- выделение разделов, пунктов, подпунктов;
- выделение приложений, таблиц и примечаний;
- построение дерева узлов;
- присвоение локаторов;
- привязку к страницам и диапазонам символов.

Результат:

- документ доступен как структурированная сущность для retrieval.

#### Шаг 9. Дедупликация и версионность

Реализовать:

- дедупликацию по обозначению;
- сравнение по хэшу;
- создание новой версии;
- переключение активной версии;
- сохранение истории.

Результат:

- база не содержит дублей активных документов и хранит историю редакций.

#### Шаг 10. Индексация

Реализовать:

- полнотекстовую индексацию;
- векторизацию узлов;
- запись в `pgvector`;
- фильтрацию по статусу и версии.

Результат:

- база готова к retrieval для Этапа 2.

#### Шаг 11. Обновление устаревших документов

Реализовать:

- проверку актуальности;
- сравнение версий;
- повторную загрузку;
- полную переобработку;
- переключение активной версии после успеха.

Результат:

- база поддерживается в актуальном состоянии.

#### Шаг 12. Интеграционные тесты и отладка

Реализовать:

- тесты на обход страниц;
- тесты на нормализацию статусов;
- тесты на парсинг карточек;
- тесты на дедупликацию;
- тесты на OCR fallback;
- тесты на обновление версий.

Результат:

- ingestion-пайплайн подтвержден тестами.

#### Шаг 13. Полный эксплуатационный end-to-end прогон

Реализовать:

- поднятие `PostgreSQL 16` с расширением `pgvector`;
- применение всех миграций на рабочей БД;
- запуск `crawl-seeds` по всем 4 исходным разделам;
- прогон worker-а до полного опустошения очереди ingestion-задач;
- сохранение полного корпуса raw-артефактов в локальном хранилище;
- снятие итоговых `ingestion-metrics` и `ingestion-report`;
- проверку фактического наполнения базы: сколько документов загружено, сколько дошло до индексации, сколько потребовало OCR;
- финальную проверку, что документы со статусом `inactive` не попали в активный индекс;
- фиксацию результата как готовой локальной базы нормативных документов для Этапа 2.

Результат:

- создана и подтверждена рабочая локальная база нормативных документов;
- Этап 1 завершен не только как кодовая реализация, но и как фактически выполненный полный ingestion корпуса;
- база готова для проверки retrieval и агентной логики на Этапе 2.

## 7. Основные решения по бизнес-логике

### 7.1. Нормализация статусов

Правило:

- в активную базу попадают только документы со статусом `active`;
- `действует`, `действующий`, `взамен` нормализуются в `active`;
- все прочие заранее известные неактивные статусы нормализуются в `inactive`;
- неизвестные статусы нормализуются в `unknown` и не попадают в активный индекс.

### 7.2. Канонический идентификатор документа

Правило:

- документ идентифицируется по нормализованному обозначению;
- обозначение должно очищаться от лишних пробелов, вариантов регистра и несущественных символов;
- если один и тот же документ найден в нескольких разделах, он должен объединяться в одну каноническую сущность.

### 7.3. Правило активации версии

Новая версия становится активной только если:

- raw-артефакты успешно загружены;
- извлечение текста завершено;
- OCR выполнен при необходимости;
- структура документа построена;
- индексы обновлены;
- не обнаружено критических ошибок.

Если любой из этапов не завершен:

- старая активная версия остается активной.

### 7.4. Правило работы с raw storage

Принятое решение:

- raw-артефакты хранятся в локальной папке проекта или в выделенной локальной директории;
- БД хранит только пути, хэши и метаданные;
- если локальная папка синхронизируется с `Google Drive`, это остается внешним механизмом и не входит в код проекта;
- переобработка всегда использует локальный файл напрямую.

### 7.5. Политика качества извлечения

Если извлечение выполнено через OCR:

- документ получает флаг `has_ocr = true`;
- сохраняется `parse_confidence`;
- если качество ниже порога, документ не удаляется, но отмечается как рискованный;
- узлы с низким качеством должны иметь соответствующую пометку для дальнейшего использования на Этапе 2.

## 8. Метрики качества

### 8.1. Метрики crawler-а

- доля успешно пройденных страниц пагинации;
- доля успешно извлеченных карточек документов;
- число страниц со сбоями;
- среднее время обхода страницы;
- число документов, пропущенных из-за ошибок.

### 8.2. Метрики загрузки

- доля успешно скачанных HTML-артефактов;
- доля успешно скачанных PDF-артефактов;
- доля успешно сохраненных raw-артефактов в локальном хранилище;
- среднее время загрузки артефакта;
- число ошибок загрузки по типам.

### 8.3. Метрики фильтрации и нормализации

- число документов со статусом `active`;
- число документов со статусом `inactive`;
- число документов со статусом `unknown`;
- доля конфликтов статуса между списком и карточкой;
- число случаев дедупликации.

### 8.4. Метрики извлечения текста

- доля документов с успешно извлеченным текстом;
- доля документов, потребовавших OCR;
- доля документов с `low_confidence_parse`;
- среднее время извлечения текста;
- среднее число структурных узлов на документ.

### 8.5. Метрики индексации

- доля документов, попавших в полнотекстовый индекс;
- доля документов, попавших в векторный индекс;
- доля версий с успешной переиндексацией;
- среднее время индексации документа.

### 8.6. Метрики актуальности

- число обнаруженных устаревших версий;
- доля успешно обновленных документов;
- среднее время полного обновления документа;
- число неудачных обновлений, после которых активная версия осталась прежней.

### 8.7. Базовые целевые ориентиры MVP

Для MVP рекомендуется ориентироваться на следующие минимальные цели:

- не менее 95% страниц сидов должны успешно обходиться;
- не менее 95% документов со статусом `active` должны успешно доходить до карточки;
- не менее 95% `active` документов должны иметь доступный raw-артефакт;
- не менее 95% `active` документов должны иметь извлеченный текст;
- не менее 95% `active` документов должны быть структурированы до уровня пунктов;
- 100% активных документов в индексе должны иметь корректный нормализованный статус;
- 0 документов со статусом `inactive` не должны попадать в активный индекс.

Этап 1 считается готовым, если:

- реализован каркас проекта и БД;

- реализована полнотекстовая и векторная индексация;
- реализован механизм обновления устаревших версий;

- подтверждено, что `inactive` документы не попали в активный индекс.

## 10. План реализации Этапа 2

### 10.1. Цель Этапа 2

Цель Этапа 2: построить расширяемую агентную систему инженерного консультирования, которая:

- принимает пользовательские запросы в формате диалога;
- самостоятельно анализирует запрос и декомпозирует его на подзадачи;
- использует локальную нормативную базу как основной слой доказательной информации;
- при необходимости подключает trusted web и open web как вспомогательные источники;
- выполняет проверку актуальности документов параллельно с подготовкой ответа;
- формирует ответ только на основе evidence-блоков с явным указанием происхождения источников;
- поддерживает несколько изолированных пользовательских сессий одновременно;
- доступна через web UI и Telegram-бота;
- допускает последующее подключение новых моделей, новых инструментов, новых источников и более автономных субагентов без переписывания ядра.

Результат этапа: работоспособная multi-session агентная платформа инженерного консультирования, использующая Stage 1 как нормативное ядро.

### 10.2. Основные принятые решения по бизнес-логике

- Архитектура этапа строится по принципу `orchestrator-first`: один основной агент-оркестратор управляет специализированными модулями, инструментами и фоновыми задачами.
- Система позиционируется не как узкий QA по нормам, а как инженерный консультант, который сам определяет, какие аспекты запроса требуют нормативного поиска, а какие требуют вспомогательной информации.
- Локальная нормативная база Stage 1 является primary evidence layer и имеет приоритет перед внешними источниками.
- Проверка актуальности документов должна выполняться неблокирующе: параллельно с подготовкой ответа, без обязательной задержки пользовательского ответа.
- Если локальная версия документа оказалась устаревшей, но refresh еще не завершен, ответ все равно допускается, но обязан явно указывать использованную локальную редакцию, факт наличия более новой версии и необходимость повторной проверки.
- Внешний поиск допускается в двух режимах: `trusted_web` и `open_web`. Любая информация из внешних источников должна маркироваться как ненормативная и требующая проверки.
- Все существенные выводы должны опираться либо на нормативные цитаты с локаторами, либо на явно маркированные внешние evidence-блоки.
- Память системы на этапе 2 ограничена текущей сессией. Межсессионная долговременная пользовательская память в объем этапа 2 не входит.
- Система должна поддерживать несколько одновременных пользовательских сессий, при этом контекст, история, tool state, кэш, фоновые задачи и временные артефакты разных сессий обязаны быть изолированы.
- Web UI и Telegram должны использовать единое прикладное ядро и одни и те же правила оркестрации, верификации и безопасности.
- Архитектура должна быть vendor-agnostic по отношению к LLM, embedding и будущим модельным сервисам.
- Контур безопасности обязателен с первого инкремента: prompt injection из пользовательского ввода, документов и web-источников должен рассматриваться как базовый архитектурный риск.

### 10.3. Принятый стек

#### 10.3.1. Backend и исполнение

- Язык: `Python 3.12`
- API-слой: `FastAPI`
- Валидация и конфигурация: `Pydantic v2`, `pydantic-settings`
- ORM и доступ к БД: `SQLAlchemy 2.x`
- Миграции: `Alembic`
- HTTP-клиент для внешних вызовов: `httpx`
- Фоновые задачи: `ARQ`
- Быстрое runtime-хранилище и coordination layer: `Redis`

#### 10.3.2. Хранение и retrieval

- Основная БД: `PostgreSQL 16`
- Векторный поиск: `pgvector`
- Полнотекстовый поиск: `PostgreSQL Full Text Search`
- Основной источник нормативных данных: БД и raw storage, созданные на Этапе 1

#### 10.3.3. Web UI и каналы доступа

- Web UI: `Next.js`, `React`, `TypeScript`
- Streaming ответа и статусов: `SSE`, при необходимости допускается `WebSocket`
- Telegram-адаптер: `aiogram 3`

#### 10.3.4. Web search и внешние источники

- Trusted sources: собственные adapters/crawlers и локальный индекс по allowlist-доменам
- Open web search: self-hosted `SearXNG`
- Paid search API в MVP не используются

#### 10.3.5. Наблюдаемость и аудит

- Instrumentation: `OpenTelemetry`
- Метрики: `Prometheus`
- Дашборды: `Grafana`
- Логи: `Loki`
- Трейсы: `Tempo` или `Jaeger`
- Аудит: собственные audit-таблицы в `PostgreSQL`

#### 10.3.6. LLM и embedding providers

Платные API допускаются только для LLM и embeddings. Архитектурно должны поддерживаться:

- `Gemini`
- `OpenAI`
- `Anthropic`
- `Qwen`
- `DeepSeek`
- локальные или self-hosted варианты через `Ollama`, `LM Studio`, `vLLM`

Использование конкретной модели должно быть конфигурируемым и не должно менять бизнес-логику оркестратора.

Отдельное уточнение по transport layer провайдеров:

- `QwenProvider` и `DeepSeekProvider` должны быть оформлены как отдельные provider-адаптеры;
- внутри адаптера допускается использование общего `OpenAI-compatible` transport, если конкретный endpoint совместим с ним;
- при необходимости адаптер должен уметь работать через native API соответствующего вендора;
- выбор transport-реализации является внутренней деталью адаптера, а не архитектурным ограничением всей системы.

#### 10.3.7. Контейнеризация и локальное развертывание

- локальная оркестрация Этапа 2 должна выполняться через `docker-compose.stage2.yml`;
- `api` и `worker` должны собираться из одного общего backend-образа Python и отличаться только командой запуска;
- `web` должен собираться отдельным образом `Next.js`;
- в `docker-compose.stage2.yml` должны быть предусмотрены профили `core`, `search`, `obs` и опционально `db`;
- профиль `core` должен включать `api`, `worker`, `web` и `Redis`;
- профиль `search` должен включать `SearXNG`;
- профиль `obs` должен включать `Prometheus`, `Grafana`, `Loki` и `Tempo` или `Jaeger`;
- профиль `db` должен быть опциональным и использоваться только для изолированных прогонов, так как основная разработка Этапа 2 может выполняться поверх уже подготовленной БД Этапа 1;
- сервисы должны иметь healthchecks, зависимости по готовности и явную схему env wiring и volume mounts.

### 10.4. Основные архитектурные решения

- Этап 2 реализуется как расширение существующего модульного монолита без раннего дробления на отдельные сервисы.
- Ядро должно строиться на собственной typed orchestration layer, а не на framework-centric логике `LangChain`, `LangGraph`, `LlamaIndex` или аналогов.
- Оркестратор обязан работать через формализованный `QueryState`, типизированные evidence-объекты и контролируемый registry инструментов.
- Провайдеры моделей, поиска и внешних интеграций должны быть инкапсулированы за абстрактными интерфейсами.
- `PostgreSQL` используется как canonical store для сессий, запросов, ответов, evidence и аудита.
- `Redis` используется как hot state layer для progress, locks, streaming coordination, TTL-кэша и фоновых задач.
- Retrieval должен быть гибридным: `FTS + vector + metadata filters + document/version constraints`.
- Проверка актуальности, refresh документов, trusted source sync и open web research должны выполняться в фоновых задачах и не блокировать основную обработку запроса.
- Все вызовы внешних источников и инструментов должны проходить через единый policy-controlled слой с логированием и аудитом.
- Системные промпты агентов не должны хардкодиться внутри агентных модулей. Они должны храниться в отдельном prompt catalog, версионироваться в репозитории и подгружаться через отдельный слой рендеринга prompt templates.
- Контур верификации должен поддерживать ограниченный `repair loop`: после первичной проверки допускается один или несколько целевых repair-pass, но только в пределах заранее заданных лимитов и со стоп-условиями при отсутствии улучшения.
- Локальное эксплуатационное окружение должно подниматься контейнерами: `api` и `worker` из общего backend-образа, `web` из отдельного frontend-образа, а инфраструктурные сервисы через `docker-compose` profiles.
- Архитектура должна допускать появление более автономных субагентов позднее, но MVP Этапа 2 не должен зависеть от распределенного multi-agent runtime.

Назначение:

Предлагаемые модули:

### 10.6. План разработки по шагам

#### Шаг 1. Базовый runtime Этапа 2

Должно быть реализовано:

- новый конфигурационный слой для Этапа 2;
- каркас `FastAPI` приложения;
- подключение `Redis` и `ARQ`;
- контейнеризированный локальный runtime для `api`, `worker`, `web` и инфраструктурных сервисов;
- базовые health/readiness endpoints;
- начальные миграции для таблиц Этапа 2;
- каркас provider registry и policy layer.

Результат шага:

- backend поднимается локально;
- `core` профиль `docker-compose.stage2.yml` поднимает `api`, `worker`, `web` и `Redis`;
- применяются миграции Этапа 2;
- фоновый worker запускается и обслуживает тестовые задачи.

#### Шаг 2. Сессии, сообщения и транспортные контракты

Должно быть реализовано:

- создание новой сессии;
- хранение истории сообщений;
- привязка запросов к `session_id` и `channel`;
- базовый SSE-streaming событий;
- единый request/response контракт для web и Telegram.

Результат шага:

- система принимает сообщения пользователя и сохраняет их в рамках сессии;
- возможен повторный запрос в рамках того же контекста.

#### Шаг 3. Абстракция моделей и provider gateway

Должно быть реализовано:

- интерфейс `ChatModelProvider`;
- интерфейс `EmbeddingProvider`;
- prompt catalog и prompt rendering layer;
- конфигурируемый выбор моделей по ролям;
- базовые адаптеры для как минимум одного cloud-провайдера и одного local/self-hosted провайдера;
- отдельные provider-адаптеры для `Qwen` и `DeepSeek` с возможностью использовать shared compatible transport или native API;
- единый формат ошибок, таймаутов и retry-policy.

Результат шага:

- orchestration и retrieval работают через абстракции, а не через vendor-specific код;
- системные промпты управляются отдельно от кода агентов;
- переключение между моделями выполняется конфигурацией.

#### Шаг 4. Нормативный retrieval поверх Stage 1

Должно быть реализовано:

- поиск по `documents`, `document_versions`, `document_nodes` и связанным таблицам Stage 1;
- гибридный retrieval `FTS + vector + metadata filters`;
- фильтрация по активной версии документа;
- сбор evidence-объекта с документом, редакцией, локатором, цитатой и freshness status.

Результат шага:

- система возвращает релевантные нормативные evidence-блоки для пользовательского запроса.

#### Шаг 5. Оркестратор и декомпозиция запроса

Должно быть реализовано:

- `QueryState`;
- анализатор запроса;
- декомпозиция на подзадачи;
- маршрутизация между нормативным retrieval, freshness и внешним research;
- запись подзадач и промежуточных результатов в БД.

Результат шага:

- сложный запрос разбивается на управляемые подзадачи;
- оркестратор работает детерминированно и прозрачно для отладки.

#### Шаг 6. Синтез ответа и формат evidence-based ответа

Должно быть реализовано:

- синтез ответа только на основе evidence;
- раздельная маркировка нормативных и ненормативных сведений;
- форматирование ссылок на документ, редакцию, локатор и цитату;
- пометка неполного покрытия вопроса.

Результат шага:

- система выдает первичный инженерный ответ с доказательной базой.

#### Шаг 7. Неблокирующая проверка актуальности и refresh

Должно быть реализовано:

- background freshness check;
- нормализованные статусы `fresh`, `stale`, `refresh_in_progress`, `refresh_failed`, `unknown`;
- запуск refresh документа без блокировки ответа;
- включение warning-блоков о stale-версии в финальный ответ.

Результат шага:

- ответ продолжает выдаваться быстро, но пользователь видит реальный статус свежести использованных документов.

#### Шаг 8. Trusted sources и open web search

Должно быть реализовано:

- allowlist доверенных доменов;
- adapters/crawlers для trusted sources;
- локальный индекс trusted sources;
- интеграция с self-hosted `SearXNG` для open web;
- нормализация внешних evidence-блоков и обязательная маркировка ненормативного происхождения.

Результат шага:

- система может закрывать пробелы в ответе через trusted web и open web без смешения этих данных с нормами.

#### Шаг 9. Контур верификации и безопасности

Должно быть реализовано:

- `citation_auditor` как гибридный `model-assisted` модуль с кодовыми инвариантами по ссылкам и локаторам;
- `coverage_auditor` как гибридный `model-assisted` модуль для проверки полноты покрытия вопроса;
- `hallucination_guard` как гибридный `model-assisted` модуль для проверки supportedness утверждений;
- `safety_guard`;
- bounded `repair loop` для исправимых замечаний verification layer;
- лимиты и стоп-условия repair loop: `max_verification_retries`, `max_total_attempts`, `max_tool_calls`, `max_time_budget`, остановка при неизменном evidence set или повторяющихся findings;
- sanitation внешнего контента;
- защита от prompt injection;
- инструментальные ограничения и policy checks;
- запись `security_events`.

Результат шага:

- система не выдает существенные тезисы без evidence;
- система либо исправляет устранимые замечания через ограниченный repair-pass, либо завершает запрос деградированным, но честно размеченным ответом;
- retrieved-контент не может подменять системные инструкции;
- нарушения фиксируются в аудите.

#### Шаг 10. Web UI

Должно быть реализовано:

- чатовый интерфейс;
- список сессий;
- streaming ответа;
- панель источников и evidence;
- индикация freshness status и источников происхождения;
- отображение ограничений и предупреждений.

Результат шага:

- пользователь может полноценно работать с системой через браузер.

#### Шаг 11. Telegram-канал

Должно быть реализовано:

- Telegram bot adapter;
- сопоставление чатов и сессий;
- прием сообщений и отправка стриминговых или поэтапных ответов;
- использование того же оркестратора и тех же правил ответа, что и в web UI.

Результат шага:

- система доступна через Telegram без форка бизнес-логики.

#### Шаг 12. Наблюдаемость, аудит и операционная диагностика

Должно быть реализовано:

- трассировка запросов и tool calls;
- базовые метрики latency, retrieval, freshness и quality;
- централизованные логи;
- audit trail по ключевым событиям;
- диагностические dashboards и readiness checks.

Результат шага:

- система наблюдаема в эксплуатации и пригодна для отладки.

#### Шаг 13. Интеграционные, нагрузочные и security-проверки

Должно быть реализовано:

- end-to-end тесты пользовательского сценария;
- тесты параллельных сессий;
- тесты stale/freshness сценариев;
- тесты open web и trusted web fallback;
- тесты prompt injection и session isolation;
- приемочный отчет Этапа 2.

Результат шага:

- зафиксировано, что функциональность Этапа 2 реально работает на готовой базе Stage 1 и соответствует критериям качества.

### 10.8. Метрики качества Этапа 2

#### 10.8.1. Метрики производительности

- `time_to_first_event`
- `time_to_first_token`
- `time_to_final_answer`
- длительность отдельных стадий: analysis, retrieval, synthesis, verification, freshness
- queue latency фоновых задач

#### 10.8.2. Метрики retrieval

- доля запросов, для которых нормативный retriever возвращает релевантный evidence в `top-k`;
- среднее число нормативных evidence-блоков на ответ;
- доля ответов, использующих только Stage 1 без внешних источников;
- доля запросов, ушедших в trusted web;
- доля запросов, ушедших в open web.

#### 10.8.3. Метрики качества ответа

- доля ответов, где каждый существенный тезис имеет ссылку на evidence;
- доля ответов с корректной маркировкой происхождения источников;
- доля ответов, помеченных как неполные;
- число ответов, отклоненных verification layer;
- число обнаруженных hallucination-like случаев.

#### 10.8.4. Метрики свежести

- доля ответов, где freshness status определен;
- число stale-документов, обнаруженных во время ответа;
- число успешно инициированных refresh-задач;
- доля refresh-задач, завершившихся успешно;
- среднее время freshness check.

#### 10.8.5. Метрики безопасности и изоляции

- число `security_events` по типам;
- число заблокированных prompt injection попыток;
- число нарушений tool policy;
- число сбоев изоляции между сессиями;
- число утечек shared state между пользователями.

#### 10.8.6. Метрики каналов доступа

- доля успешных запросов через web UI;
- доля успешных запросов через Telegram;
- среднее время ответа по каждому каналу;
- число оборванных streaming-сессий.

#### 10.8.7. Базовые целевые ориентиры MVP

Для MVP Этапа 2 рекомендуется ориентироваться на следующие минимальные цели:

- 100% существенных нормативных тезисов в финальном ответе должны иметь документ, редакцию, локатор и цитату либо явную пометку о внешнем происхождении;
- 100% ответов с использованием trusted web или open web должны содержать предупреждение о ненормативном происхождении и необходимости проверки;
- 100% ответов, в которых обнаружена stale-версия, должны содержать предупреждение о свежести;
- `0` сбоев изоляции между пользовательскими сессиями;
- `0` случаев, когда retrieved-контент исполняется как инструкция;
- не менее 95% web-запросов и не менее 95% Telegram-запросов должны завершаться без внутренней ошибки;
- для warm-path нормативного запроса должен быть обеспечен приемлемый streaming-старт ответа без ожидания завершения freshness refresh.

### 10.9. Критерии готовности Этапа 2

Этап 2 считается готовым, если:

- реализован backend-слой `FastAPI` для работы с пользовательскими сессиями и запросами;
- реализован `Redis + ARQ` контур для фоновых задач;
- реализовано контейнеризированное локальное развертывание через `docker-compose.stage2.yml` для `api`, `worker`, `web` и инфраструктурных профилей;
- реализованы минимальные таблицы и миграции Этапа 2;
- оркестратор принимает пользовательский запрос и декомпозирует его на подзадачи;
- нормативный retrieval работает поверх базы Этапа 1 и возвращает evidence на уровне пунктов и подпунктов;
- ответ формируется только из evidence-блоков и содержит ссылки на документ, редакцию, локатор и цитату;
- freshness check выполняется параллельно с подготовкой ответа и не блокирует выдачу;
- при stale-версии система корректно предупреждает пользователя и инициирует refresh;
- реализован trusted web search по allowlist-доменам;
- реализован open web search через self-hosted `SearXNG`;
- ответ явно разграничивает нормативные и ненормативные источники;
- реализован web UI в формате чата со streaming ответа;
- реализован доступ через Telegram-бота;
- контекст, tool state, кэш, временные артефакты и background processing изолированы между сессиями;
- реализованы verification layer и security layer;
- реализованы аудит, логи, метрики и трейсы;
- есть приемочные интеграционные тесты Этапа 2;
- есть отчет по итоговой проверке качества и готовности Этапа 2 на реальной базе Stage 1.
